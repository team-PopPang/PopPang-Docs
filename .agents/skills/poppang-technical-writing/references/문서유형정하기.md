# Step 1. 문서 유형 정하기

문서를 잘 쓰려면 문장을 고치기 전에 먼저 문서 유형을 정해야 합니다.  
문서 유형이 정해지면 독자가 기대하는 구조, 예시 방식, 설명의 깊이, 문장의 톤을 한 방향으로 맞출 수 있습니다.

기술 문서는 크게 네 가지로 나눌 수 있습니다.

- 학습 중심 문서
- 문제 해결 중심 문서
- 참조 문서
- 설명 문서

핵심은 "내가 무엇을 쓰고 싶은가"보다 "독자가 이 문서를 왜 여는가"를 먼저 파악하는 것입니다.

## AI와 함께 쓰기

문서 유형을 고르기 어려우면 Step 1 전용 리뷰어처럼 사용합니다.

- 입력:
  - 문서 목표
  - 독자 수준
  - 프로젝트 상황
  - 이미 있는 초안 또는 메모
  - 추가 제약사항
- 출력:
  - 가장 적합한 문서 유형 1개
  - 그 유형을 추천한 이유
  - 필요한 경우에만 보조 유형 1개
  - 작성할 때 꼭 지켜야 할 포인트

한 번에 여러 방향을 늘어놓기보다, 가장 적합한 방향 하나를 먼저 정하는 데 집중합니다.

## 추천 선택지

사용자에게 문서 유형을 직접 고르게 할 때는 아래 순서를 사용합니다.

1. 시작하기 가이드
2. 튜토리얼
3. 문제 해결 가이드
4. 참조 문서
5. 깊은 이해를 위한 설명 문서
6. 추천해줘

`추천해줘`는 항상 마지막에 둡니다.

## 소개

문서 유형은 독자의 목적을 기준으로 고릅니다.

- 처음 배우는 사람은 학습 흐름이 필요합니다.
- 지금 막힌 사람은 해결 절차가 필요합니다.
- 이미 맥락을 아는 사람은 정확한 정보 조각이 필요합니다.
- 기술의 배경과 원리를 이해하려는 사람은 설명이 필요합니다.

문서 하나에 모든 목적을 다 넣으려고 하면 독자는 어디에 집중해야 할지 알기 어려워집니다.  
먼저 주된 목적 하나를 정하고, 부족한 부분은 보조 섹션으로만 섞는 편이 좋습니다.

## 문서 유형을 고르기 전에 먼저 답할 질문

### 1. 독자는 이 문서를 펼친 직후 무엇을 원하나

- 빠르게 시작하고 싶다
- 하나를 따라 만들고 싶다
- 지금 발생한 문제를 해결하고 싶다
- 특정 옵션이나 API를 확인하고 싶다
- 개념과 원리를 깊게 이해하고 싶다

### 2. 독자는 이미 무엇을 알고 있나

- 아무것도 모르는 입문자
- 기본 흐름은 아는 사용자
- 특정 문제만 해결하면 되는 사용자
- API 이름 정도는 아는 사용자

### 3. 문서를 다 읽은 뒤 독자가 해야 할 행동은 무엇인가

- 설치한다
- 따라 만든다
- 오류를 해결한다
- 구현에 바로 적용한다
- 의사결정을 내린다

### 4. 이 문서에서 가장 중요한 것은 무엇인가

- 학습 경험
- 문제 해결 속도
- 정확성과 완전성
- 배경과 맥락

이 네 질문에 답하면 어떤 문서 유형이 맞는지 대부분 정리됩니다.

## 학습을 위한 문서

학습 중심 문서는 독자가 새로운 기술이나 도구를 익히도록 돕는 문서입니다.  
핵심은 "이 문서를 따라 하면 무엇을 할 수 있는지"를 독자가 선명하게 상상할 수 있게 만드는 것입니다.

### 언제 선택하나

- 독자가 주제를 처음 접한다
- 설치, 초기 설정, 첫 실행이 중요하다
- 따라 하면서 감을 잡아야 한다
- 작은 성공 경험이 필요하다

### 이 유형에서 가장 중요한 것

- 문서가 끝났을 때 독자가 할 수 있는 일을 먼저 보여주기
- 중간에 막히지 않도록 사전 요구사항을 분명히 적기
- 단계별 흐름을 작게 끊기
- 예제를 실제로 실행 가능한 상태로 유지하기

### 세부 유형

#### 시작하기 가이드

가장 빠른 입문에 집중합니다.

- 목표: "일단 해보게 만들기"
- 길이: 짧아도 괜찮음
- 예시: 설치, 첫 실행, 최소 예제

이 유형은 모든 것을 설명하려 하지 않습니다.  
독자가 가장 빨리 "작동했다"를 경험하게 하는 데 집중합니다.

#### 튜토리얼

하나를 따라 만들면서 이해하게 하는 문서입니다.

- 목표: "완성 과정을 경험하게 만들기"
- 길이: 시작하기 가이드보다 길어질 수 있음
- 예시: 간단한 앱, 기능 하나 구현, 실제 흐름 따라 만들기

튜토리얼은 지식 전달보다 학습 경험 설계가 더 중요합니다.

### 언제 고르지 않는가

- 오류 해결이 핵심이면 학습 문서보다 문제 해결 문서가 맞습니다.
- API 옵션 목록을 보여주는 게 목적이면 참조 문서가 맞습니다.
- 배경 설명이 중심이면 설명 문서가 맞습니다.

### 꼭 들어가야 할 요소

1. 목표
2. 사전 요구사항
3. 단계별 진행
4. 실행 가능한 예제
5. 최종 결과 확인
6. 자주 막히는 문제

### 작성 원칙

- 쉬운 예제에서 시작해 점점 확장합니다.
- 단계마다 무엇을 하는지와 왜 하는지를 함께 적습니다.
- 독자가 결과를 확인할 기준을 제공합니다.
- 단계가 너무 크면 세분화합니다.

### 안 좋은 문서 냄새

- 목표가 모호하다
- 설치와 설정이 빠져 있다
- 예제가 실행되지 않는다
- 배경 설명이 너무 길어서 시작이 늦다
- 결과 확인 방법이 없다

### 예시 상황

- Flutter를 처음 설치하고 앱을 실행하는 방법
- React 프로젝트를 만들고 첫 화면을 띄우는 방법
- CLI 도구를 설치하고 기본 명령을 실행하는 방법

## 문제 해결을 위한 문서

문제 해결 중심 문서는 독자가 지금 겪고 있는 문제를 빠르게 해결하도록 돕는 문서입니다.  
핵심은 "이 문서를 보면 바로 해결할 수 있는가"입니다.

### 언제 선택하나

- 오류 메시지나 비정상 동작을 해결해야 한다
- 특정 작업 목표를 달성하는 절차가 필요하다
- 배경 설명보다 적용 가능한 해결책이 중요하다

### 문제 해결 문서 안에서도 구분해야 할 것

#### 트러블슈팅 문서

문제의 원인과 해결책을 정리하는 문서입니다.

- 예: `Cannot read property of undefined` 오류 해결
- 초점: 증상, 원인, 해결 순서, 확인 방법

#### How-to 가이드

특정 작업을 수행하는 방법을 안내하는 문서입니다.

- 예: Webpack에 플러그인 추가하기
- 초점: 목표 달성 절차

#### 에러 코드/메시지 설명

오류의 뜻을 찾아보는 사전형 문서입니다.

- 예: 특정 에러 코드의 의미와 원인
- 초점: 정의, 원인, 관련 링크

문제가 생겼다고 해서 항상 트러블슈팅 문서가 맞는 것은 아닙니다.  
독자가 원하는 것이 "원인 분석"인지, "작업 수행 절차"인지 구분해야 합니다.

### 이 유형에서 가장 중요한 것

- 문제가 무엇인지 한 문장으로 정의하기
- 언제 발생하는지 재현 조건을 적기
- 바로 적용 가능한 해결 절차 제공하기
- 해결 후 정상 상태를 확인하는 방법 적기

### 꼭 들어가야 할 요소

1. 문제 정의
2. 증상 또는 재현 조건
3. 원인
4. 해결 절차
5. 환경별 차이
6. 해결 후 확인 방법

### 작성 원칙

- 에러 메시지는 가능한 그대로 보여줍니다.
- 복붙 가능한 명령어와 코드 예제를 제공합니다.
- 해결책이 왜 통하는지도 짧게 설명합니다.
- 운영체제, 버전, 환경에 따라 달라지는 조건을 함께 적습니다.

### 안 좋은 문서 냄새

- 원인 설명 없이 해결책만 던진다
- 해결 순서가 모호하다
- 예제가 실행 불가능하다
- 해결 후 무엇이 정상인지 안 알려준다
- 배경 설명이 길어서 핵심 조치가 묻힌다

### 예시 상황

- React에서 `undefined` 관련 오류를 해결하는 방법
- 인증 실패 시 헤더 설정을 점검하는 절차
- 특정 플러그인을 프로젝트에 통합하는 방법

## 참조를 위한 문서

참조 문서는 특정 기술, 도구, 시스템에 대한 정확하고 완전한 정보를 빠르게 찾게 해주는 문서입니다.  
핵심은 정확성, 완전성, 검색 용이성입니다.

### 언제 선택하나

- 함수, 옵션, 속성, 이벤트를 찾아봐야 한다
- 이미 기본 사용법은 알고 있다
- 필요한 조각 정보를 빠르게 확인하고 싶다

### 이 유형에서 가장 중요한 것

- 구조가 일관되어야 합니다.
- 누락 없이 정확해야 합니다.
- 독자가 필요한 항목으로 바로 점프할 수 있어야 합니다.

### 꼭 들어가야 할 요소

1. 개요
2. 시그니처 또는 핵심 인터페이스
3. 매개변수/속성 표
4. 반환값
5. 사용 예제
6. 주의사항과 제한사항

### 작성 원칙

- 모든 항목이 같은 순서를 따르도록 씁니다.
- 예제는 짧고 바로 적용 가능해야 합니다.
- 용어, 표기, 표 구조를 문서군 전체에서 통일합니다.
- 탐색 가능한 제목을 씁니다.

### 언제 고르지 않는가

- 독자가 개념부터 이해해야 하면 설명 문서가 먼저입니다.
- 처음 배우는 입문자라면 학습 문서가 먼저입니다.
- 특정 문제를 해결하는 흐름이라면 문제 해결 문서가 더 낫습니다.

### 안 좋은 문서 냄새

- 속성 설명이 빠져 있다
- 어떤 값이 허용되는지 모호하다
- 예제보다 설명문이 지나치게 길다
- 페이지마다 표 구조가 다르다

### 예시 상황

- `fs.readFile`의 `options` 파라미터 설명
- 컴포넌트 props 목록
- CLI 옵션과 기본값 문서

## 깊은 이해를 위한 문서

설명 문서는 독자가 기술이나 개념을 깊게 이해하도록 돕는 문서입니다.  
핵심은 배경과 맥락을 충분히 설명하고, 의사결정 과정을 명확하게 공유하는 것입니다.

### 언제 선택하나

- 사용법보다 원리와 이유가 중요하다
- 등장 배경과 해결하려는 문제가 중요하다
- 대안 비교나 설계 판단 기준이 필요하다
- 기술의 철학과 한계를 설명해야 한다

### 이 유형에서 가장 중요한 것

- "왜 이런 기술이 필요한가"를 먼저 설명하기
- 단순 사용법보다 맥락과 원리를 중심에 두기
- 비교와 트레이드오프를 분명히 보여주기

### 꼭 들어가야 할 요소

1. 개념 소개
2. 등장 배경
3. 해결하려는 문제
4. 작동 원리
5. 대안 비교
6. 실제 활용 사례

### 작성 원칙

- 배경과 맥락을 앞에 둡니다.
- 의사결정 이유를 숨기지 않습니다.
- 장단점과 한계를 함께 적습니다.
- 필요한 경우 표, 그림, 흐름도를 사용합니다.

### 안 좋은 문서 냄새

- 원리 설명 없이 결론만 말한다
- 비교 기준이 없다
- 장점만 말하고 한계는 숨긴다
- 배경이 부족해서 왜 중요한지 모르겠다

### 예시 상황

- JavaScript의 `this` 동작 원리 설명
- 이벤트 소싱이 필요한 배경과 장단점
- SSR과 CSR의 차이와 선택 기준

## 문서 유형을 섞어야 할 때

현실의 좋은 문서는 한 가지 유형만 갖지 않을 때도 많습니다.  
다만 "한 페이지의 주된 목적"은 여전히 하나여야 합니다.

### 섞어도 좋은 경우

- 시작하기 가이드 마지막에 짧은 FAQ를 붙인다
- How-to 가이드에서 세부 옵션은 참조 문서로 분리한다
- 설명 문서 안에 개념 확인용 짧은 예시를 넣는다

### 섞으면 안 좋은 경우

- 입문 문서 안에 API 레퍼런스를 길게 넣는다
- 트러블슈팅 문서 앞부분에 긴 배경 설명을 넣는다
- 설명 문서인데 절차 중심으로만 쓴다

기본 원칙은 간단합니다.  
문서 안에 다른 유형의 요소를 조금 섞을 수는 있지만, 독자가 이 페이지를 펼친 이유를 흐리게 만들면 안 됩니다.

## 실습하기

독자의 질문을 보고 문서 유형을 고르는 연습을 합니다.

### 예시 1

질문: JavaScript에서 `this` 키워드가 왜 상황마다 다르게 동작하는지 알고 싶어요.

- 추천 유형: 깊은 이해를 위한 설명 문서
- 이유: 사용법보다 원리, 평가 방식, 등장 배경을 이해하는 것이 핵심입니다.

### 예시 2

질문: Webpack 설정 파일에 특정 플러그인을 추가하는 방법을 알고 싶어요.

- 추천 유형: 문제 해결 가이드(How-to)
- 보조 유형: 참조 문서
- 이유: 지금 필요한 것은 작업 절차이고, 세부 옵션은 별도 참조 문서가 더 적합합니다.

### 예시 3

질문: React에서 `Cannot read property of undefined` 오류를 해결하고 싶어요.

- 추천 유형: 문제 해결 가이드(트러블슈팅)
- 이유: 증상, 원인, 해결 순서, 정상 동작 확인이 핵심입니다.

### 예시 4

질문: `fs.readFile`의 `options` 파라미터에 어떤 값을 넣을 수 있는지 알고 싶어요.

- 추천 유형: 참조 문서
- 이유: 정확한 속성, 허용값, 기본값, 예제를 빠르게 찾아야 합니다.

### 예시 5

질문: Flutter를 처음 설치하고 앱을 띄우는 흐름을 알고 싶어요.

- 추천 유형: 시작하기 가이드
- 이유: 빠른 입문과 첫 성공 경험이 핵심입니다.

## 유형별 템플릿 예시

### 1. 학습 중심 문서

```md
# [문서 제목]

## 목표
[이 문서를 마치면 독자가 할 수 있는 일]

## 사전 요구사항
- [필요한 도구]
- [필요한 계정 또는 버전]

## 빠른 개요
[전체 흐름을 2~4문장으로 요약]

## 단계별 가이드
### 1. [첫 단계]
[무엇을 왜 하는지]

### 2. [다음 단계]
[실행 예제]

## 결과 확인
[정상 결과]

## 자주 막히는 문제
[FAQ]
```

### 2. 문제 해결 중심 문서

```md
# [문제 또는 작업 제목]

## 문제 정의
[어떤 문제인지 또는 어떤 작업을 하려는지]

## 증상 또는 목표
[현재 상태]

## 원인 또는 배경
[왜 이런 문제가 생기는지]

## 해결 방법
### 1. [첫 번째 조치]
[명령어 또는 코드]

### 2. [두 번째 조치]
[필요 시 대안]

## 확인 방법
[정상 동작 기준]
```

````md
## 예제
[짧은 예제]

````

## 추천 규칙

문서 유형을 자동 추천할 때는 아래 규칙을 우선합니다.

- 설치, 첫 실행, 최소 성공 경험이 중요하면 `시작하기 가이드`
- 따라 만들며 배우는 경험이 중요하면 `튜토리얼`
- 특정 작업을 끝내야 하면 `문제 해결 가이드`
- 오류를 해결해야 하면 `문제 해결 가이드`
- API, 옵션, 속성 조회가 목적이면 `참조 문서`
- 배경, 원리, 비교가 목적이면 `설명 문서`

판단이 애매하면 "독자가 문서를 열자마자 가장 먼저 원하는 것"을 기준으로 정합니다.

## 적용용 출력 형식

문서 유형을 추천할 때는 아래 형식을 권장합니다.

- 추천 문서 유형
- 추천 이유
- 보조 유형이 필요하면 1개만 추가
- 이 유형에서 반드시 넣어야 할 섹션
- 피해야 할 구조

## 체크리스트

- 독자의 목표가 분명한가
- 주된 문서 유형이 하나로 정리되는가
- 이 문서에서 가장 중요한 경험이 학습인지, 해결인지, 참조인지, 설명인지 분명한가
- 문서 유형에 맞는 구조를 바로 떠올릴 수 있는가
- 다른 유형의 요소가 핵심 목적을 흐리지 않는가
