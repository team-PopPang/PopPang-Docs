# 문체 및 프롬프트 원칙

이 문서는 팝팡 전용 기술 문서 스킬의 문체 잠금 파일이다.
기존 `technical-writing.dev`는 해요체와 합쇼체를 섞어 쓰기도 하지만, 이 스킬은 토스식 일관성을 더 강하게 적용한다.

## 1. 토스식 문체 원칙

- 설명은 짧고 분명하게 쓴다.
- 한 문장에 하나의 메시지만 둔다.
- 불필요한 완곡어법을 줄인다.
- 독자가 바로 이해할 수 있는 단어를 쓴다.
- 메타 담화를 줄이고 바로 핵심으로 들어간다.
- 문장을 예쁘게 꾸미기보다 빨리 이해되게 쓴다.

## 2. 어미 사용 원칙

- 같은 결과물 안에서는 어미를 함부로 섞지 않는다.
- 기본값은 `-요.` 체다.
- 체크리스트, 규칙, 판정 문장, 스킬 내부 지침은 `-다.` 체를 써도 된다.
- 같은 문단 안에서 `-요.`와 `-다.`를 섞지 않는다.
- source prompt를 그대로 옮길 때는 원문 어미를 유지한다.

## 3. 결과물 유형별 권장 톤

### 사용자에게 보여주는 가이드 초안

- 기본: `-요.` 체
- 목적: 부담 없이 읽히고 빠르게 이해되게 한다

### 스킬 내부 규칙, 체크리스트, 실행 절차

- 기본: `-다.` 체
- 목적: 판단 기준을 또렷하게 만든다

### 프롬프트 원문

- 원문 유지
- 목적: 각색으로 의미나 뉘앙스가 바뀌지 않게 한다

## 4. 프롬프트 보존 원칙

- source page에 프롬프트가 있으면 그대로 사용한다.
- 맞춤법, 말투, 어순, 항목 순서를 임의로 손보지 않는다.
- 앞뒤 설명을 덧붙일 수는 있지만 프롬프트 본문은 수정하지 않는다.
- 프롬프트 일부만 재구성한 요약본을 원문인 것처럼 제시하지 않는다.

## 5. 추측 금지 원칙

- 버전, 수치, API 세부값, 운영체제 차이, 정책, 프로세스는 근거 없이 지어내지 않는다.
- 필요한 정보가 없으면 먼저 부족한 정보를 분명히 말한다.
- 꼭 진행해야 하면 최소한의 가정만 두고, 가정임을 명시한다.
- 문체를 맞춘다는 이유로 내용 의미를 바꾸지 않는다.

## 6. 문장 다듬기 우선순위

1. 의미 유지
2. 정보 정확성
3. 독자 이해 속도
4. 문장 길이 축소
5. 톤 일관성

문장을 더 짧게 만들 수 있더라도 의미가 흔들리면 줄이지 않는다.
