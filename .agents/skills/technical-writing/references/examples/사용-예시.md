# Technical Writing 스킬 사용 가이드

이 문서는 `technical-writing` 스킬을 실제로 어떻게 요청하면 좋은지 보여주는 사용 가이드입니다.  
핵심은 "문서 유형 정하기 -> 정보 구조 만들기 -> 문장 다듬기" 흐름을 한 번에 적용하게 요청하는 것입니다.

## 이 스킬로 할 수 있는 것

이 스킬은 아래 작업을 처리할 수 있습니다.

- 기존 문서에 맞는 문서 유형 추천
- 기존 문서의 구조 진단과 재구성
- 기존 문장의 기술 문서화
- 초안이 없는 상태에서 새 문서 생성
- 초보자/중급자/숙련자에 맞춘 톤 조정
- 한 페이지에 너무 많은 내용이 섞인 문서 분리 제안
- 같은 주제를 시작하기 가이드, 튜토리얼, 문제 해결 문서, 참조 문서, 설명 문서 중 어떤 형식으로 써야 할지 추천
- "추천해줘" 옵션으로 문서 유형까지 알아서 고르게 하기

## 언제 쓰면 좋은가

아래 상황이면 이 스킬이 잘 맞습니다.

- 문서는 있는데 너무 장황하거나 흐름이 이상하다
- 어떤 문서 유형으로 써야 할지 애매하다
- 빠르게 입문용 문서를 만들어야 한다
- 오류 해결 문서를 더 실전적으로 만들고 싶다
- API 설명을 참조 문서답게 정리하고 싶다
- 번역투, 메타 담화, 명사형 표현이 많은 문장을 자연스럽게 바꾸고 싶다

## 요청할 때 함께 주면 좋은 정보

가능하면 아래 정보를 같이 주면 결과가 더 좋아집니다.

- 독자 수준
  - 예: Flutter를 처음 배우는 초급 개발자
- 문서 목표
  - 예: 설치부터 첫 실행까지 끝내게 하기
- 프로젝트 상황
  - 예: macOS 기준, VS Code 사용
- 원하는 문서 유형
  - 예: 튜토리얼로 써줘
- 이미 있는 초안
  - 예: 메모, 초벌 문서, 회의록

정보가 적어도 스킬은 동작하지만, 독자와 목표가 분명할수록 문서 품질이 좋아집니다.

## 가장 자주 쓰는 요청 패턴

### 1. 기존 문서를 전체적으로 다듬기

가장 기본적인 요청입니다.  
문서 유형 추천, 구조 개선, 문장 개선을 한 번에 수행하게 만들 수 있습니다.

```text
아래 문서를 technical-writing 스킬로 다듬어줘.
독자는 Flutter를 처음 배우는 개발자야.
문서 유형을 먼저 추천하고, 구조와 문장도 함께 고쳐줘.

[문서 본문]
```

이 요청을 하면 보통 아래 순서로 진행합니다.

1. 문서 유형 추천
2. 구조 문제 진단
3. 정보 구조 재정리
4. 문장 다듬기
5. 최종 개선본 제시

### 2. 문서 유형 선택지부터 보여주기

사용자가 어떤 형식으로 써야 할지 먼저 고르고 싶을 때 유용합니다.

```text
이 문서에 맞는 유형을 먼저 골라줘.
선택지는 시작하기 가이드, 튜토리얼, 문제 해결 가이드, 참조 문서, 깊은 이해를 위한 설명 문서, 추천해줘 순서로 보여줘.
그다음 네가 추천한 방향으로 진행해줘.

[문서 본문]
```

이 요청은 아래 상황에 특히 좋습니다.

- 한 문서 안에 학습, 참조, 설명이 섞여 있을 때
- 어떤 형식이 맞는지 팀 안에서도 의견이 갈릴 때
- 먼저 문서 방향부터 합의하고 싶을 때

### 3. 추천해줘까지 포함해서 알아서 진행하게 하기

사용자가 문서 유형을 직접 고르기 싫다면 이렇게 요청할 수 있습니다.

```text
아래 주제에 맞는 문서 유형을 추천해줘.
선택지는 보여주되, 마지막에 추천해줘도 넣어줘.
나는 추천해줘를 고른 걸로 보고 바로 진행해줘.

[주제 또는 문서 초안]
```

이 패턴은 빠르게 작업을 시작할 때 가장 편합니다.

### 4. 구조만 집중적으로 손보기

문장은 아직 건드리지 않고 목차와 흐름만 먼저 잡고 싶을 때 사용합니다.

```text
이 문서의 문장 표현은 일단 그대로 두고,
technical-writing 스킬로 정보 구조만 다시 짜줘.
한 페이지에 주제가 너무 많이 섞였는지도 봐줘.

[문서 본문]
```

이 요청을 하면 보통 아래를 돌려줍니다.

- 현재 구조 진단
- 핵심 목표가 하나인지 판정
- 개요 개선안
- 재구성된 목차
- 분리/병합 제안

### 5. 문장만 집중적으로 다듬기

구조는 괜찮고 표현만 기술 문서답게 바꾸고 싶을 때 사용합니다.

```text
구조는 유지하고 문장만 다듬어줘.
번역투, 메타 담화, 명사형 표현을 줄이고
자연스러운 한국어 기술 문서 문장으로 바꿔줘.

[문단 또는 문서]
```

이 요청은 아래 상황에 특히 좋습니다.

- 사람이 급하게 쓴 초안이 있다
- 영어 문서를 번역한 느낌이 강하다
- 군더더기 표현이 많다

### 6. 새 문서 만들기

본문 초안이 없어도 주제만 있으면 새 문서를 생성할 수 있습니다.

```text
flutter 사용법 문서 만들어줘.
독자는 모바일 앱을 처음 만드는 초급 개발자야.
추천 문서 유형을 먼저 정하고, 적합한 형식으로 초안을 써줘.
```

보통 이런 흐름으로 생성합니다.

1. 추천 문서 유형 선택
2. 독자와 목표 정리
3. 개요 작성
4. 추천 목차 작성
5. 본문 초안 생성
6. 자주 막히는 지점 또는 체크리스트 추가

### 7. 시작하기 가이드로 강제하기

입문 문서는 가장 짧고 빠르게 성공 경험을 주는 것이 중요할 때가 많습니다.

```text
이 주제를 시작하기 가이드 형식으로 써줘.
설치, 첫 실행, 최소 성공 경험 중심으로 짧고 선명하게 구성해줘.

[주제]
```

예를 들면:

- Flutter 설치 후 첫 앱 실행
- React 프로젝트 생성 후 개발 서버 실행
- CLI 도구 설치 후 첫 명령 실행

### 8. 튜토리얼로 강제하기

따라 하면서 하나를 완성하게 만들고 싶을 때 적합합니다.

```text
이 주제를 튜토리얼 형식으로 써줘.
독자가 직접 따라 하면서 결과물을 만들 수 있게 단계별로 구성해줘.

[주제]
```

이 패턴은:

- 간단한 기능 만들기
- 샘플 앱 완성하기
- 하나의 흐름을 손으로 익히게 하기

에 잘 맞습니다.

### 9. 문제 해결 가이드로 만들기

오류 해결이나 특정 작업 수행 절차를 문서화하고 싶을 때 좋습니다.

```text
이 내용을 문제 해결 가이드로 바꿔줘.
문제 정의, 원인, 해결 절차, 확인 방법이 분명하게 보이게 구성해줘.

[문서 초안 또는 오류 설명]
```

이 요청은 특히:

- 에러 메시지 정리
- 트러블슈팅 문서
- How-to 절차 문서

에 잘 맞습니다.

### 10. 참조 문서 형식으로 바꾸기

옵션, 속성, API 설명처럼 정확성과 빠른 탐색이 중요할 때 씁니다.

```text
이 내용을 참조 문서 형식으로 다시 써줘.
개요, 시그니처, 옵션 설명, 반환값, 예제, 주의사항 순서로 정리해줘.

[기존 설명]
```

이 요청은:

- 함수 문서
- props 문서
- 설정 옵션 문서
- CLI 플래그 문서

에 적합합니다.

### 11. 설명 문서 형식으로 바꾸기

사용법보다 배경과 원리를 이해시키는 것이 더 중요할 때 쓰면 좋습니다.

```text
이 주제를 깊은 이해를 위한 설명 문서로 써줘.
왜 필요한지, 어떤 문제를 해결하는지, 어떤 대안이 있는지 중심으로 설명해줘.

[주제]
```

예를 들면:

- JavaScript의 `this`
- SSR과 CSR 비교
- 이벤트 소싱이 필요한 이유

### 12. 문서군으로 나누기

한 문서 안에 너무 많은 것이 섞여 있으면, 한 번에 여러 페이지 구조를 제안하게 만들 수도 있습니다.

```text
이 문서는 한 페이지에 내용이 너무 많이 섞여 있어.
technical-writing 스킬로 문서군 구조로 나눠줘.
어떤 페이지를 새로 만들어야 하는지도 제안해줘.

[문서 본문]
```

이 요청을 하면 보통:

- 상위 개요 페이지
- 하위 시작하기 문서
- 참조 문서
- 트러블슈팅 문서

처럼 분리 방향까지 제안할 수 있습니다.

## 출력 결과로 기대할 수 있는 것

좋은 요청을 넣으면 보통 아래 요소가 포함된 결과를 받게 됩니다.

- 추천 문서 유형
- 추천 이유
- 독자와 목표
- 개요 또는 문서 진단
- 개선된 목차
- 다듬은 본문
- 자주 막히는 지점
- 체크리스트

## 잘 먹히는 요청 방식

아래처럼 요청하면 결과가 더 안정적입니다.

### 좋은 방식

```text
아래 문서를 다듬어줘.
독자는 React 초급 개발자야.
문서 유형을 추천한 뒤, 구조와 문장을 함께 개선해줘.
용어는 기존 표기를 최대한 유지해줘.
```

### 아쉬운 방식

```text
문서 좀 좋아지게 해줘
```

너무 짧은 요청도 동작은 하지만, 독자와 목표가 빠지면 결과 방향이 흔들릴 수 있습니다.

## 요청할 때 자주 붙이면 좋은 옵션

아래 문장을 덧붙이면 원하는 방향을 더 정확히 맞출 수 있습니다.

- 독자는 [초급/중급/숙련] 개발자야
- 문서 유형을 먼저 추천해줘
- 선택지는 꼭 먼저 보여줘
- 추천해줘 옵션도 마지막에 넣어줘
- 구조만 먼저 고쳐줘
- 문장만 먼저 다듬어줘
- 예시는 실행 가능한 수준으로 써줘
- 배경 설명은 짧게, 실습은 길게 써줘
- 참조 문서처럼 간결하게 써줘
- 튜토리얼처럼 따라 하기 쉽게 써줘

## 빠르게 복붙해서 쓰는 요청 모음

### 기존 문서 개선

```text
아래 문서를 technical-writing 스킬로 다듬어줘.
독자는 [독자].
문서 유형을 먼저 추천하고, 구조와 문장까지 한 번에 고쳐줘.

[문서]
```

### 새 문서 생성

```text
[주제] 문서 만들어줘.
독자는 [독자].
문서 유형을 먼저 추천하고, 가장 적합한 형식으로 초안을 써줘.
```

### 구조만 개선

```text
이 문서의 문장은 그대로 두고 구조만 개선해줘.
핵심 목표가 하나인지도 같이 봐줘.

[문서]
```

### 문장만 개선

```text
구조는 유지하고 문장만 다듬어줘.
번역투, 메타 담화, 명사형 표현을 줄여줘.

[문서]
```

### 문서 유형부터 선택

```text
이 주제에 맞는 문서 유형을 먼저 골라줘.
선택지는 시작하기 가이드, 튜토리얼, 문제 해결 가이드, 참조 문서, 깊은 이해를 위한 설명 문서, 추천해줘 순서로 보여줘.
추천 이유도 짧게 적어줘.

[주제 또는 문서]
```

## 이 스킬이 특히 잘하는 것

- 막연한 초안을 기술 문서답게 바꾸는 일
- 문서 목표에 맞는 유형을 골라주는 일
- 한 페이지에 섞인 내용을 분리하는 일
- 초보자를 위한 시작 문서를 빠르게 만드는 일
- 장황하고 번역투인 문장을 자연스럽게 바꾸는 일

## 이 스킬을 쓸 때 기억하면 좋은 한 줄

가장 좋은 요청은 "독자", "문서 목표", "원하는 깊이", "기존 초안 유무"를 함께 주는 요청입니다.
