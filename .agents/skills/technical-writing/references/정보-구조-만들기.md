# Step 2. 정보 구조 만들기

정보 구조는 문서의 내용을 독자가 작은 노력으로 이해하게 만드는 설계입니다.  
기술 문서는 쓰는 사람의 머릿속 순서가 아니라, 읽는 사람이 목표를 달성하는 순서로 배치해야 합니다.

핵심은 다음 여섯 가지입니다.

- 한 페이지에서는 하나만 다루기
- 가치를 먼저 제공하기
- 효과적인 제목 쓰기
- 개요 빠트리지 않기
- 예측 가능하게 하기
- 자세히 설명하기

## 소개

문서 구조를 다듬을 때 가장 먼저 볼 것은 "이 페이지가 독자에게 무엇 하나를 해결해 주는가"입니다.  
그다음에 개요, 제목, 흐름, 배경 설명, 세부 정보를 순서대로 맞춥니다.

문서 구조가 좋으면:

- 독자가 필요한 정보를 더 빨리 찾을 수 있고
- 문서를 끝까지 읽을지 빠르게 판단할 수 있고
- 같은 주제의 다른 문서도 더 쉽게 이해할 수 있습니다

## AI와 함께 쓰기

Step 2는 문서를 다시 배열하는 단계입니다. 구조 전용 리뷰어처럼 사용합니다.

- 입력:
  - 문서 초안
  - 현재 목차
  - 문서 목표
  - 독자 수준
  - 현재 문제점
- 출력:
  - 구조 진단
  - 핵심 목표 판정
  - 개요 개선안
  - 재구성 목차
  - 분리/병합 제안

여러 개선 방향을 나열하는 것보다, 모두 반영한 하나의 좋은 구조안을 만드는 데 집중합니다.

## 구조를 보기 전에 먼저 수집할 정보

문서 구조를 바꾸기 전에 아래를 먼저 적어두면 판단이 빨라집니다.

- 이 문서를 여는 독자의 대표 질문
- 이 문서를 다 읽으면 독자가 할 수 있어야 하는 일
- 지금 문서에서 가장 중요한 섹션
- 지금 문서에서 없어도 되는 섹션
- 다른 문서로 빼야 할 주제

이 질문에 답하지 않으면 구조를 바꾸는 대신 위치만 섞게 되는 경우가 많습니다.

## 한 페이지에서는 하나만 다루기

한 페이지에는 하나의 핵심 목표만 두는 것이 좋습니다.  
한 문서에 너무 많은 질문과 개념을 한꺼번에 넣으면 독자는 무엇을 얻어가야 하는지 파악하기 어렵습니다.

### 왜 중요한가

- 독자가 핵심을 더 빨리 이해할 수 있습니다.
- 필요한 정보를 찾는 시간이 줄어듭니다.
- 큰 문서를 개요 페이지와 상세 페이지로 나누기 쉬워집니다.

### 적용 기준

- 제목 깊이가 `####`(H4) 이상으로 계속 깊어지면 문서를 나눌지 검토합니다.
- 서로 다른 질문에 답하는 섹션이 섞여 있으면 분리합니다.
- 설치, 개념 설명, API 표, 오류 해결이 한 문서에 다 들어 있으면 쪼갤 가능성이 큽니다.
- 하나의 페이지가 하나의 동작, 하나의 개념, 하나의 문제를 설명하는지 확인합니다.

### 판단 질문

- 이 페이지를 읽고 독자가 얻어가야 할 한 가지는 무엇인가
- 문서 제목을 한 문장으로 바꿨을 때 자연스러운가
- 섹션 일부를 다른 문서로 빼면 전체 이해가 더 쉬워지는가

### 안 좋은 예

- "React 시작하기" 문서 안에 설치, 상태 관리 철학, API 레퍼런스, 오류 해결, 배포 방법이 모두 섞여 있다

### 더 나은 방향

- 입문은 시작하기 가이드
- 상태 관리 철학은 설명 문서
- API는 참조 문서
- 오류 해결은 트러블슈팅 문서
- 배포는 How-to 가이드

### 적용 액션

- 핵심이 다른 섹션은 분리합니다.
- 상위 개요 페이지를 두고 하위 상세 페이지로 연결합니다.
- 각 페이지 제목이 정확히 하나의 질문에 답하게 만듭니다.

## 가치를 먼저 제공하기

기능이나 세부 설정을 먼저 설명하기보다, 독자가 얻는 가치를 먼저 보여주는 것이 좋습니다.  
독자는 "이게 왜 필요한가", "이 문서를 읽으면 무엇이 달라지는가"를 먼저 알고 싶어 합니다.

### 왜 중요한가

- 독자가 계속 읽을 이유를 즉시 제공합니다.
- 핵심 메시지를 더 빨리 이해하게 합니다.
- 세부 정보가 나와도 맥락을 잃지 않습니다.

### 적용 기준

- 문서 도입부에서 해결되는 문제를 먼저 말합니다.
- 이 문서를 읽으면 얻는 결과를 먼저 말합니다.
- 세부 옵션, 내부 구현, 부가 조건은 뒤로 보냅니다.
- 기능 목록보다 독자 효용을 앞에 둡니다.

### 안 좋은 흐름

기능 설명 -> 옵션 설명 -> 세부 예외 -> 왜 필요한지 설명

### 좋은 흐름

어떤 문제를 해결하는지 -> 무엇을 얻게 되는지 -> 어떻게 쓰는지 -> 세부 옵션과 예외

### 적용 액션

- 첫 문단에 "이 문서가 해결하는 문제"를 넣습니다.
- 개요 또는 목표 섹션에 "읽고 나면 할 수 있는 일"을 적습니다.
- 사소한 제약사항은 뒤 섹션으로 이동합니다.

## 효과적인 제목 쓰기

제목은 문서의 핵심 내용을 간결하고 명확하게 전달해야 합니다.  
독자가 스캔하면서 빠르게 내용을 파악하고, 검색으로도 쉽게 찾을 수 있어야 합니다.

### 적용 기준

- 핵심 키워드를 포함합니다.
- 제목은 30자 이내를 기본 기준으로 삼습니다.
- 같은 수준의 제목은 같은 문체와 리듬을 유지합니다.
- 가능하면 평서문으로 씁니다.
- 제목만 읽어도 다룰 내용이 드러나야 합니다.

### 좋은 제목의 특징

- 질문에 가까운 표현을 씁니다.
- 너무 넓거나 추상적이지 않습니다.
- 페이지 역할이 드러납니다.

### 안 좋은 제목

- 소개
- 기타
- 고급 기능
- 팁

이런 제목은 내용이 무엇인지 바로 떠오르지 않습니다.

### 더 나은 제목

- API 요청 최적화하기
- SSR 환경에서 쿠키 읽기
- 인증 헤더 설정하기
- 업로드 실패 원인 점검하기

### 적용 액션

- 제목만 따로 읽었을 때도 흐름이 보이게 만듭니다.
- 형제 섹션 제목을 나란히 놓고 톤이 맞는지 확인합니다.
- 개념, 절차, 예시, 확인 섹션이 섞여 있다면 제목 체계를 다시 맞춥니다.

## 개요 빠트리지 않기

개요는 문서의 핵심 내용을 요약하고, 독자가 문서를 읽기 전에 내용을 빠르게 이해하도록 돕습니다.  
좋은 개요가 있으면 독자는 문서를 끝까지 읽을지, 지금 필요한 문서인지 빠르게 판단할 수 있습니다.

### 왜 중요한가

- 문서의 가치를 첫 화면에서 전달할 수 있습니다.
- 독자가 읽기 전에 전체 흐름을 파악할 수 있습니다.
- 배경 설명보다 먼저 핵심을 볼 수 있습니다.

### 개요에 들어갈 것

- 이 문서가 다루는 대상
- 이 문서를 읽으면 할 수 있는 일
- 필요한 사전 조건
- 문서의 큰 흐름

### 개요 작성 팁

- 2~4문단 또는 짧은 목록으로 끝냅니다.
- 기술적 배경 설명보다 독자 효용을 먼저 적습니다.
- 문서 목표를 한 문장으로 요약합니다.

### 안 좋은 개요

- 역사부터 길게 설명한다
- 목표가 없다
- 이 문서를 왜 읽어야 하는지 안 보인다

### 더 나은 개요 예시

이 문서에서는 OAuth 로그인 흐름을 설정하는 방법을 설명합니다.  
문서를 마치면 인증 제공자 등록, 리디렉션 URI 설정, 콜백 처리까지 한 번에 구성할 수 있습니다.  
시작하기 전에 클라이언트 ID와 클라이언트 시크릿이 준비되어 있어야 합니다.

### 적용 액션

- 문서 첫 화면에 개요를 배치합니다.
- 개요만 읽어도 "내게 필요한 문서인가"를 판단할 수 있게 만듭니다.
- 상세 설명은 개요 뒤로 미룹니다.

## 예측 가능하게 하기

문서는 독자가 쉽게 탐색하고 필요한 정보를 빠르게 찾을 수 있도록 예측 가능해야 합니다.  
일관된 구조는 인지 부담을 줄이고, 문서가 많아질수록 탐색 비용을 크게 낮춥니다.

### 왜 중요한가

- 독자가 원하는 정보를 더 빨리 찾습니다.
- 새 문서를 읽을 때도 구조를 예상할 수 있습니다.
- 팀이 함께 문서를 유지보수하기 쉬워집니다.

### 적용 기준

- 같은 수준의 제목은 같은 형식을 따릅니다.
- 개념 -> 절차 -> 예시 -> 확인 같은 흐름을 일관되게 유지합니다.
- 용어를 같은 방식으로 씁니다.
- 표, 코드 블록, 체크리스트 위치도 통일합니다.
- 비슷한 성격의 페이지는 같은 순서를 유지합니다.

### 안 좋은 예

- 어떤 페이지는 개요가 있고 어떤 페이지는 없다
- 어떤 페이지는 예시가 먼저 나오고 어떤 페이지는 마지막에만 나온다
- 같은 수준의 제목이 평서문과 명사구로 뒤섞여 있다

### 더 나은 방향

- 같은 문서군은 같은 뼈대를 공유합니다.
- 비슷한 페이지는 독자가 같은 위치에서 같은 종류의 정보를 찾게 만듭니다.

### 적용 액션

- 문서군 공통 섹션 순서를 정합니다.
- 제목 네이밍 규칙을 정합니다.
- 반복되는 표와 예제 형식을 템플릿화합니다.

## 자세히 설명하기

독자는 이 문서를 처음 읽는 사람일 수 있습니다.  
새로운 개념을 생략하거나, 기능이 어떤 조건에서 어떻게 동작하는지 설명하지 않으면 독자는 문장을 추론으로 메워야 합니다. 그 순간 문서는 읽기 어려워집니다.

### 왜 중요한가

- 학습 곡선을 줄일 수 있습니다.
- 기능을 더 정확하게 이해하게 합니다.
- 잘못된 적용과 오해를 줄일 수 있습니다.

### 적용 기준

- 새로운 개념이 등장하면 1~2문장으로 정의합니다.
- 왜 알아야 하는지, 어디에서 쓰는지 함께 적습니다.
- 입력값, 상태, 시간에 따라 결과가 달라지면 그 조건을 밝힙니다.
- 예제와 설명이 따로 놀지 않게 연결합니다.

### 안 좋은 예

이 서비스는 이벤트 소싱 방식으로 상태를 관리합니다.

- 이벤트 소싱이 무엇인지 설명이 없습니다.
- 독자는 왜 이 문장에서 이벤트 소싱을 알아야 하는지 알 수 없습니다.

### 더 나은 방향

- 이벤트 소싱이 무엇인지 짧게 정의합니다.
- 왜 이 방식이 필요한지 설명합니다.
- 이 문서에서 이벤트 소싱이 어떤 동작 차이를 만드는지 연결합니다.

### 적용 액션

- 새 용어는 첫 등장 위치에서 정의합니다.
- "왜 필요한가", "언제 쓰는가", "어떤 조건에서 달라지는가"를 함께 적습니다.
- 배경 설명은 핵심을 흐리지 않을 만큼만, 하지만 추론이 필요 없을 만큼은 충분히 제공합니다.

## 구조 개선 워크플로우

Step 2를 실제로 적용할 때는 아래 순서를 권장합니다.

1. 문서의 핵심 목표를 한 문장으로 적는다
2. 지금 목차에서 핵심 목표와 무관한 섹션을 표시한다
3. 개요를 먼저 쓴다
4. 제목만 읽어도 흐름이 보이도록 다시 정렬한다
5. 세부 설명, 배경 설명, 예시를 적절한 위치로 옮긴다
6. 다른 페이지로 분리해야 할 내용을 분리한다

## 빠른 진단 예시

### 현재 상태

- 제목: `API 요청 최적화`
- 본문: 성능 기법, 속도 측정, 오류 처리, 캐싱, 인증 설정이 섞여 있음

### 진단

- 한 페이지에서 목표가 여러 개입니다.
- 성능 최적화와 오류 처리가 섞여 있습니다.
- 개요가 없어서 얻는 가치가 첫 화면에서 드러나지 않습니다.

### 개선 방향

- `API 요청 최적화하기`
- `API 응답 속도 측정하기`
- `API 요청 오류 해결하기`

처럼 분리하거나, 최소한 상위 개요 아래에서 분기해야 합니다.

## 적용용 출력 형식

문서 구조를 개선할 때는 아래 형식을 권장합니다.

- 현재 구조 진단
- 핵심 목표가 하나인지 판정
- 개요 개선안
- 재구성된 목차
- 분리/병합 제안
- 제목 수정 제안
- 빠진 배경 설명 제안

## 체크리스트

- 한 페이지에 목표가 하나인가
- 문서 첫 화면에서 가치를 먼저 전달하는가
- 제목이 짧고 구체적인가
- 개요가 문서 목표를 분명하게 보여주는가
- 비슷한 문서끼리 구조가 예측 가능한가
- 새로운 개념에 필요한 배경 설명이 붙어 있는가
- 세부 정보가 핵심 정보보다 앞에 오지 않는가
