# Step 3. 문장 다듬기

문장을 잘 다듬는다는 것은 단순히 짧게 줄이는 일이 아닙니다.  
의미는 유지하면서도 더 명확하고, 더 자연스럽고, 더 읽기 쉽게 만드는 일입니다.

기술 문서는 정보 구조가 좋아도 문장이 흐리면 전달력이 급격히 떨어집니다.  
그래서 Step 3은 마지막 손질이 아니라, 독자가 실제로 이해하게 만드는 핵심 단계입니다.

## 소개

이 단계에서 보는 핵심 원칙은 다섯 가지입니다.

- 문장의 주체를 분명하게 하기
- 필요한 정보만 남기기
- 구체적으로 쓰기
- 자연스러운 한국어 표현 쓰기
- 일관되게 쓰기

여기에 더해, 불필요한 메타 담화를 줄이는 원칙도 함께 적용합니다.

## AI와 함께 쓰기

Step 3는 문장 전용 편집기처럼 사용합니다.

- 입력:
  - 문장 또는 문단
  - 문서 전체 초안
  - 독자 수준
  - 유지해야 할 용어
- 출력:
  - 다듬은 문장
  - 바꾼 이유
  - 필요하면 짧은 대안 1개

여러 개선안을 늘어놓기보다, 모든 원칙을 반영한 가장 좋은 한 버전을 만드는 데 집중합니다.

## 다듬기 순서

문장을 다듬을 때는 아래 순서가 가장 안정적입니다.

1. 문장의 주체가 누구인지 먼저 본다
2. 한 문장에 생각이 몇 개 들어 있는지 본다
3. 모호한 명사형 표현을 동사형으로 바꾼다
4. 번역체와 어색한 한자어를 정리한다
5. 같은 개념이 다른 말로 반복되는지 본다

이 순서를 따르면 "짧아졌는데 오히려 의미가 흐려지는" 실수를 줄일 수 있습니다.

## 문장의 주체를 분명하게 하기

기술 문서를 읽는 독자는 주로 개발자나 사용자입니다.  
또 실제로 작업을 수행하는 사람도 대개 독자입니다. 따라서 기술 문서에서는 독자나 개발자가 행동의 주체가 되도록 쓰는 편이 더 자연스럽고 명확합니다.

### 왜 중요한가

- 누가 무엇을 해야 하는지 바로 이해할 수 있습니다.
- 문장이 덜 기계적으로 느껴집니다.
- 책임과 행동이 선명해집니다.

### 적용 기준

- 도구, 기술, 시스템을 행동의 주체로 과하게 쓰지 않습니다.
- 가능하면 독자나 개발자가 무엇을 하는지 중심으로 씁니다.
- 수동형보다 능동형을 우선합니다.

### 안 좋은 예

- 이 라이브러리는 데이터베이스 초기화를 수행합니다.
- 인증 토큰이 갱신됩니다.

### 더 나은 예

- 개발자는 이 라이브러리로 데이터베이스를 초기화할 수 있습니다.
- 서버가 만료된 토큰을 받으면 새 토큰을 발급합니다.

### 체크 질문

- 문장을 읽고 누가 행동하는지 바로 보이는가
- 수동형을 능동형으로 바꿨을 때 더 자연스러워지는가
- 도구를 의인화해서 쓰고 있지 않은가

## 필요한 정보만 남기기

문장은 짧고 간결할수록 읽기 쉽습니다.  
한 문장에는 하나의 생각만 담는 것이 기본입니다.

### 왜 중요한가

- 긴 문장은 핵심을 묻히게 합니다.
- 여러 생각이 섞이면 잘못 이해하기 쉽습니다.
- 읽는 속도가 느려지고 기억하기 어려워집니다.

### 적용 기준

- 한 문장에는 하나의 핵심만 둡니다.
- 긴 문장은 둘 이상으로 나눕니다.
- 반복되는 설명을 줄입니다.
- 문단 안에서도 중심 메시지를 하나로 유지합니다.

### 안 좋은 예

이 API를 호출할 때 요청 헤더와 인증 정보를 포함해야 정상적으로 응답을 받을 수 있으며, 일부 환경에서는 타임아웃 설정도 필요할 수 있고, 응답 형식도 함께 확인해야 합니다.

### 더 나은 예

이 API를 호출할 때는 요청 헤더와 인증 정보를 포함하세요.  
일부 환경에서는 타임아웃 설정이 필요합니다.  
호출한 뒤에는 응답 형식도 함께 확인하세요.

### 메타 담화도 함께 줄이기

메타 담화는 핵심 정보를 전달하지 않고 말에 대한 말만 덧붙이는 표현입니다.

- 앞서 설명했듯이
- 이제 알아보겠습니다
- 여러분도 아시다시피
- 다음과 같이 진행할 수 있습니다

이 표현은 꼭 필요하지 않다면 지우고 바로 정보로 들어가는 편이 좋습니다.

## 구체적으로 쓰기

모호하거나 장황한 표현은 독자의 이해를 방해합니다.  
기술 문서에서는 독자가 지금 무엇을 해야 하는지, 무엇이 달라지는지, 어떤 조건에서 결과가 바뀌는지를 바로 알 수 있어야 합니다.

### 적용 기준

- 명사 대신 동사를 우선합니다.
- 모호한 표현 대신 직접적인 표현을 씁니다.
- 누가, 무엇을, 어디에, 어떻게 하는지 드러냅니다.
- 가능하면 결과와 조건을 함께 적습니다.

### 1. 명사 대신 동사를 사용하기

#### 안 좋은 예

- 코드 최적화 진행 후 배포 수행이 필요합니다.
- MongoDB 연결 정보 설정 및 초기화가 필요합니다.

#### 더 나은 예

- 코드를 최적화한 후 배포하세요.
- MongoDB에 연결할 호스트와 포트를 설정하고 데이터베이스를 초기화합니다.

### 2. 모호한 표현 대신 명확한 표현 쓰기

#### 안 좋은 예

- 설정 파일을 변경하면 기존 설정이 영향을 받을 수도 있습니다.
- 일부 브라우저에서 정상적으로 동작하지 않을 가능성이 있습니다.

#### 더 나은 예

- 설정 파일을 변경하면 기존 설정이 삭제됩니다.
- Internet Explorer에서는 정상적으로 동작하지 않습니다.

### 3. 필요한 맥락을 문장 안에 넣기

#### 안 좋은 예

- 설정한 뒤 실행하세요.

#### 더 나은 예

- `.env` 파일에 API 키를 설정한 뒤 개발 서버를 실행하세요.

### 체크 질문

- 문장에서 실제 행동이 보이는가
- 모호한 가능성 표현이 남아 있지 않은가
- 독자가 추가 맥락 없이도 이해할 수 있는가

## 자연스러운 한국어 표현 쓰기

기술 문서에서는 정확하고 간결한 표현이 중요합니다.  
불필요한 한자어, 번역체 표현, 잡초 표현을 줄이면 문장이 더 직관적이고 읽기 쉬워집니다.

### 적용 기준

- 불필요한 한자어를 줄입니다.
- 번역체 표현을 자연스럽게 고칩니다.
- 영어식 명사형 표현보다 동사형 표현을 씁니다.
- 조사, 어순, 호응이 자연스러운지 확인합니다.

### 1. 불필요한 한자어 줄이기

#### 안 좋은 예

로그 파일을 삭제하는 작업을 수행합니다.

#### 더 나은 예

로그 파일을 삭제합니다.

### 2. 번역체 표현 다듬기

영어 문장은 명사형 표현이 자연스러울 때가 많지만, 한국어는 동사 중심으로 풀어 쓸 때 훨씬 읽기 쉽습니다.

#### 안 좋은 예

- API 키를 이용한 사용자 인증 처리가 완료된 후, 데이터베이스 접속 설정 진행이 가능합니다.
- 시스템 모니터링 수행을 통해 서버 성능 측정 작업을 실시하게 됩니다.
- 보안 정책 업데이트 진행 후 시스템 재시작 처리를 수행해야 합니다.

#### 더 나은 예

- API 키로 사용자를 인증한 후, 데이터베이스에 접속하도록 설정할 수 있습니다.
- 시스템을 모니터링해서 서버 성능을 측정합니다.
- 보안 정책을 업데이트한 후 시스템을 재시작합니다.

- 아래와 같이 확인이 가능합니다.

### 체크 질문

- 한국어로 말할 때도 이렇게 표현할까
- 한자어를 빼면 의미가 더 또렷해지는가
- 번역체 어순이 남아 있지 않은가

## 일관되게 쓰기

같은 개념을 여러 방식으로 표현하면 독자는 혼란을 느끼고 검색도 어려워집니다.  
기술 문서의 신뢰감은 용어 일관성에서 크게 올라갑니다.

### 적용 기준

- 공식 기술 용어를 따릅니다.
- 같은 개념은 한 문서 안에서 같은 표현으로 유지합니다.
- 약어는 처음 등장할 때 풀어쓴 뒤 사용합니다.
- 외래어 표기는 업계 관례를 고려합니다.
- 대소문자 표기까지 일관되게 맞춥니다.

### 1. 공식 기술 용어 사용하기

#### 안 좋은 예

K8을 사용하면 애플리케이션 배포가 쉬워집니다.

#### 더 나은 예

쿠버네티스(Kubernetes)를 사용하면 애플리케이션 배포가 쉬워집니다.

### 2. 같은 개념을 같은 말로 유지하기

#### 안 좋은 예

파일을 추가하려면 '파일 선택' 버튼을 클릭하세요. 파일을 첨부한 후 '저장'을 누르면 업로드가 완료됩니다. 필요한 경우 파일을 다시 넣을 수 있습니다.

#### 더 나은 예

파일을 업로드하려면 '파일 선택' 버튼을 클릭하세요. 파일을 업로드한 후 '저장'을 누르면 업로드가 완료됩니다. 필요한 경우 파일을 다시 업로드할 수 있습니다.

### 3. 약어는 먼저 풀어쓰기

#### 안 좋은 예

이 기능은 SSR을 지원합니다.

#### 더 나은 예

이 기능은 서버 사이드 렌더링(Server-Side Rendering, SSR)을 지원합니다.

### 체크 질문

- 같은 개념이 여러 단어로 섞여 있지 않은가
- 대소문자와 공식 표기가 맞는가
- 약어를 먼저 풀어썼는가

## 실전 다듬기 예시

### 원문

이 기능을 사용하기 위해서는 인증 설정 진행 후 요청 헤더 값 세팅을 수행해야 하며 일부 상황에서는 응답값이 다르게 내려올 수 있습니다.

### 문제점

- 명사형 표현이 많음
- 한 문장에 생각이 여러 개 섞여 있음
- 누가 무엇을 해야 하는지 흐림
- "일부 상황"이 모호함

### 개선본

이 기능을 사용하려면 먼저 인증을 설정하세요. 그다음 요청 헤더 값을 넣으세요. 테스트 환경에서는 응답 형식이 운영 환경과 다를 수 있습니다.

## 적용용 출력 형식

문장을 다듬을 때는 아래 형식을 권장합니다.

- 원문 요약
- 다듬은 문장 또는 문단
- 바꾼 이유
  - 명확성
  - 간결성
  - 자연스러움
  - 일관성
- 필요하면 짧은 대안 1개

## 체크리스트

- 문장의 주체가 분명한가
- 한 문장에 하나의 생각만 담았는가
- 메타 담화가 불필요하게 들어가지 않았는가
- 명사형 표현보다 동사형 표현을 썼는가
- 모호한 표현 대신 구체적인 표현을 썼는가
- 한국어 어순과 표현이 자연스러운가
- 같은 개념을 같은 용어로 유지했는가
