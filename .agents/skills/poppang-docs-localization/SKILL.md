# PopPang 문서 한글화

외부 원문의 구조와 기술적 정확성을 유지하면서 PopPang-Docs에 완전한 한국어 Rspress 문서를 추가한다. 요약문이 아니라, 독자가 원문과 같은 정보를 한국어로 읽을 수 있게 만드는 작업에 집중한다.

## 작업 순서

### 1. 범위와 문서 구조 확인

1. 프로젝트의 `AGENTS.md`, `rspress.config.ts`, 대상 섹션의 `_meta.json`, `docs/_nav.json`을 읽는다.
2. 사용자가 지정한 섹션, 페이지 제목, 원문 URL을 우선한다.
3. 제목이나 `description`이 없으면 원문의 제목과 도입부를 근거로 제안한다. 페이지 위치나 새 섹션 여부가 불분명하고 결과에 큰 영향을 준다면 그 값만 짧게 확인한다.
4. 새 문서의 `description`은 한국어로 50~160자 사이에서 작성한다.

### 2. 원문 전체 확보

1. 원문 페이지 전체를 읽는다. 검색 결과의 발췌문이나 일부 화면만으로 번역하지 않는다.
2. JavaScript로 렌더링되는 문서라면 원본 저장소의 Markdown·MDX·DocC 소스, 공식 API 문서, 다운로드 가능한 원문 순으로 찾는다.
3. 최신 정보가 중요한 문서라면 공식 원문과 버전·업데이트 시점을 확인한다.
4. 번역을 시작하기 전에 제목, 코드 블록, 표, 안내문, 내부·외부 링크를 확인한다.

### 3. 누락 없는 번역 작성

1. 제목, 문단, 목록, 표, 이미지 설명, 참고·팁·경고, 코드 블록, 링크를 모두 유지한다. 내용을 축약하거나 예제를 생략하지 않는다.
2. 제목과 본문, 목록의 설명은 자연스러운 한국어로 번역한다. 대상 섹션의 기존 문체를 따르고, 기준이 없으면 `-요.` 체를 일관되게 쓴다.
3. 코드 식별자, API 이름, 명령어, 파일 경로, URL, 버전, 플랫폼 이름, 숫자는 바꾸지 않는다. 코드 주석은 동작에 영향을 주지 않는 범위에서 번역할 수 있다.
4. `Optional`, `enum`, `reducer`, `store`처럼 코드와 밀접한 용어는 원문 표기를 유지하거나 최초 한 번 한국어 설명을 덧붙인다. 같은 문서에서는 표기를 통일한다.
5. 원문의 DocC 문법, 예를 들어 이중 백틱 API 참조나 `<doc:…>` 링크가 Rspress에서 동작하지 않으면 API 경로를 인라인 코드로 유지한다. 원문 정보를 지우거나 임의의 링크를 만들지 않는다.
6. 문서 첫머리에 원문 제목과 URL을 연결해 출처를 남긴다.

### 4. Rspress에 통합

1. `rspress.config.ts`의 `root`에서 문서 루트를 확인한다.
2. 새 페이지에는 `title`과 `description` frontmatter를 넣는다.
3. 대상 폴더의 `_meta.json`에 slug를 추가한다.
4. 새 섹션이면 상위 `_meta.json`에 `dir-section-header`를 추가한다. 사용자가 섹션 추가를 요청했거나 사이트에서 바로 찾을 수 있어야 한다면 `docs/_nav.json`에도 메뉴를 연결한다.
5. 같은 주제의 기존 문서에서 새 페이지로 이어져야 한다면 링크를 추가한다. 관련 없는 문서 구조는 바꾸지 않는다.

### 5. 완전성 및 빌드 검증

1. 원문과 번역본의 Markdown 제목 수, 코드 펜스 수, 참고·팁·경고 수를 비교한다. 차이가 있으면 의도적인 Rspress 변환인지 확인하고, 누락이라면 보완한다.
2. 추가한 페이지의 제목, 설명, 사이드바 항목, 상단 메뉴, 내부 앵커와 원문 링크를 확인한다.
3. `git diff --check`을 실행한다.
4. `AGENTS.md`가 요구하는 포맷과 린트 명령을 실행하고 `npm run build`로 정적 생성까지 검증한다. 전역 포매터가 관련 없는 기존 파일을 바꾸면 사용자 변경은 보존하고, 도구가 만든 무관한 변경만 `apply_patch`로 되돌린다.

## 완료 보고

완료 시 아래를 간단히 알린다.

- 추가한 문서와 내비게이션 파일
- 원문 URL
- 원문과 일치하는 제목·코드 블록·안내문 검증 결과
- 실행한 포맷, 린트, 빌드 결과
